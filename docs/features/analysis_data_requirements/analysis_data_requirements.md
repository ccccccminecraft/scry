# 分析精度向上のために必要なデータ

## 概要

現在のScryログは「何をプレイしたか」は記録されているが、**「なぜそのプレイをしたか」「その結果ライフはどう動いたか」** が抜けている。以下に分析したい問いと、それに必要なデータを対応させて整理する。

---

## 1. ライフ推移データ（優先度：最高）

**欲しい分析**
- 何ターン目に詰め切れば勝てたか
- どのスペルが実質的に最もライフを削ったか

**現状の問題**

アクションログにダメージの数値が入っていないため、たとえば Slickshot Show-Off が特定のターンに何点通したのかを計算できない。Mutagenic Growth を保持すべきだった場面の判断も、実際のライフ総量が見えないと評価できない。

**必要なデータ**

各ターン終了時点での両プレイヤーのライフ総量。

```
例）
turn: 3
CCCCCC_life: 16
opponent_life: 11
```

---

## 2. サイドボード投入・抜き内容（優先度：高）

**欲しい分析**
- ゲーム2・3でのサイドボーディングが適切だったか
- 投入したカードが実際に機能したか

**現状の問題**

現在のログにはGame 1→2→3の間に何をサイドインしたかが記録されていない。そのため「Consign to Memory を4枚入れた結果どうなったか」が検証できない。

**必要なデータ**

各ゲーム開始時のデッキリスト（サイドボード後）、またはサイドイン/アウトの差分。

```
例）
game: 2
sideboard_in: [Consign to Memory x4, Spell Pierce x2]
sideboard_out: [Violent Urge x3, Lava Dart x2, Preordain x1]
```

---

## 3. 墓地の内容（優先度：高）

**欲しい分析**
- デリリアム達成のタイミングが最適だったか
- DRC や Violent Urge の条件が何ターン目に揃っていたか

**現状の問題**

墓地にあるカードタイプの種類数が追跡されていないため、Delirium が実際に何ターン目に達成されたかが後から判定できない。

**必要なデータ**

墓地の内容（またはカードタイプ別の枚数）のターンごとスナップショット。

```
例）
turn: 3
graveyard_types: [creature, instant, land]  # → delirium未達成
turn: 4
graveyard_types: [creature, instant, sorcery, land]  # → delirium達成
```

---

## 4. 手札情報（優先度：中）

**欲しい分析**
- マリガンすべき手札だったか
- 特定ターンに打てたのに打たなかったカードがあったか

**現状の問題**

ログにはcastされたカードしか記録されないため、手札に抱えたまま使わなかったカードが見えない。たとえばゲーム2でPreordainを連打しながら詰め切れなかった場面で、手札に何が残っていたのかが不明。

**必要なデータ**

各ターン開始時の手札枚数と、理想的にはドロー内容のログ。

```
例）
turn: 4
hand_size: 3
drawn: Lightning Bolt
```

---

## 5. 盤面のパワー/タフネス実値（優先度：中）

**欲しい分析**
- 攻撃を通すべきだったか、ブロックすべきだったか
- Prowessトリガー後の実際のサイズが何点のアタックに相当したか

**現状の問題**

attackログはあるが、そのターンにどれだけのProwessが乗っていたかが数値として残っていない。たとえば Monastery Swiftspear でのattackがProwessで何点になっていたのかを計算できない。

**必要なデータ**

各アタック・ブロック時のクリーチャーのP/T実値。

```
例）
turn: 3
attacker: Monastery Swiftspear
power: 3  # prowess x2 適用後
toughness: 3
```

---

## 6. 選択肢の分岐ログ（優先度：低）

**欲しい分析**
- 「あのターンにAではなくBを選んでいたら？」という反実仮想の評価

**現状の問題**

これは自動取得が最も難しい項目で、現状は対戦後にプレイヤーが手動で記録するしかない。たとえば Preordain でどちらのカードを底に送ったか、Expressive Iteration でどのカードを追放したかはログに残らない。

**必要なデータ**

モーダル呪文・スクライ・ルーティングの選択内容。

```
例）
turn: 1
cast: Preordain
scry_top: Dragon's Rage Channeler
scry_bottom: Mountain
decision: kept_top
```

|--------|-----------|---------|---------|-----------|---------------|
| 最高 | ライフ推移 | ✅ 存在・部分実装済 | ❌ ログになし | 低 | 勝敗の「あと何点」が定量化できる |
| 高 | サイドボード投入内容 | ✅ 存在・未保存 | ❌ ログになし | 低 | ゲーム2以降の分析精度が根本的に変わる |
| 高 | 墓地のカードタイプ数 | ⚠️ 再構築可 | ❌ ログになし | 中 | Deliriumデッキの分析に必須 |

## ログ調査結果（2026-03-25 実施）

パーサーコード（`log_parser.py` / `gre_parser.py` / `surveil_parser.py`）およびMTGAサンプルログ（`Player.log`）を実際に調査した結果を記録する。

### MTGA ログの特性

MTGAのログはGREプロトコルに基づくJSONフォーマットで、フェーズ・ステップ単位の詳細なイベントが記録されている。情報量はMTGOより大幅に多い。

**ライフ推移**: `AnnotationType_ModifiedLife` アノテーションと `gameStateMessage.players[i].lifeTotal` フィールドにより、変動のタイミングと現在値の両方が取得できる。`gre_parser.py` はすでに `_handle_life_change` で変動イベントを検知・detail に記録しているが、DBへの保存は未実装。

**サイドボード差分**: 各ゲーム開始時の `submitDeckReq` に `deckCards`（メイン）と `sideboardCards`（サイド）の grpId 配列が含まれる。`gre_parser.py` はこれを `deck_grp_ids` / `sideboard_grp_ids` として保持済みだが、ゲーム間の差分計算とDBへの保存は未実装。カード名への変換は import 時に `mtga_cards` テーブルから解決可能。

**墓地状態**: ゾーン間移動を表す `ZoneTransfer` アノテーションから墓地行きイベントを追跡できる。ただし「現在の墓地スナップショット」として直接提供されるわけではなく、ZoneTransfer イベントの累積から再構築する必要がある。カードタイプの判定には card_cache または Scryfall の `type_line` が必要。

**手札・ドロー**: `ZoneTransfer` の category `"Draw"` をもとに draw アクションが保存済み（grpId が解決できた場合のみカード名付き）。ターン開始時の手札枚数は `zones[].type = "ZoneType_Hand"` の `objectInstanceIds` 配列長から取得可能だが未保存。

**P/T実値**: GREプロトコルはP/Tを明示的に送信しない。アタック・ブロック宣言には instanceId のみが含まれ、そのターンの修正値（Prowessなど）を含む実際のP/Tはログから復元できない。Scryfall の静的P/Tは取得可能だが戦闘時の修正値は不可。

**スクライ・ルーティング選択**: GREログに選択内容は記録されない。根本的にログレベルでの取得は不可能。

### MTGO ログの特性

MTGOのログはテキスト形式で、主要アクション（プレイ・アタック・ドロー枚数など）のみが記録されている。情報量はMTGAより大幅に少ない。

ライフ、サイドボード、墓地状態に関するデータはログに含まれていない。ドローは枚数のみ記録（カード名なし）。P/T・スクライ選択も同様に存在しない。

### 結論

ライフ推移・サイドボード差分・墓地状態・手札枚数の4項目は**MTGAのみ**で実装可能。MTGOについては技術的に対応困難なため、これらの機能はMTGA専用として実装し、MTGOインポート分はデータなしとして扱う方針が現実的。

P/T実値とスクライ選択はMTGA・MTGO双方でログに存在せず、実装スコープ外とする。
